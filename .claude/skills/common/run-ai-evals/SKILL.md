# run-ai-evals

## この skill を使う場面
- AI 機能のリリース前品質チェック
- プロンプト変更後の品質回帰検証
- ai-engineer が品質を定量的に評価するとき

## 入力前提
- 評価データセット（質問-期待回答のペアまたは評価基準）
- LLM サービスが利用可能

## 実行手順

### 1. 評価データセットの準備
```json
[
  {
    "id": "eval-001",
    "input": "PostgreSQL のインデックスの種類を教えて",
    "expectedStructure": { "type": "list", "minItems": 3 },
    "expectedBehavior": "技術的に正確なリストを返す",
    "category": "factual"
  },
  {
    "id": "eval-002",
    "input": "",
    "expectedBehavior": "空入力に対して適切なエラーメッセージを返す",
    "category": "boundary"
  }
]
```

### 2. 評価観点
| 観点 | 評価方法 | 合格基準 |
|------|---------|---------|
| 構造 | JSON スキーマ準拠、必須フィールド存在 | 100% |
| 失敗時動作 | エラー時のフォールバック表示 | 100% |
| 引用 | ソース情報の含有（RAG 使用時） | 80% |
| 再試行 | リトライ後の成功率 | 90% |
| 境界条件 | 空入力、長文、特殊文字 | 100% |
| レスポンス時間 | p95 | < 10秒 |

### 3. 評価の実行
```bash
# 評価スクリプトの実行
npx ts-node scripts/run-ai-evals.ts --dataset=evals/dataset.json --output=evals/results.json
```

### 4. 結果の集計
```markdown
## AI 評価結果
- 実施日: YYYY-MM-DD
- モデル: gpt-4o
- データセット: N件

| 観点 | 合格率 | 基準 | 判定 |
|------|--------|------|------|
| 構造 | 100% | 100% | ✅ |
| 失敗時動作 | 100% | 100% | ✅ |
| 引用 | 85% | 80% | ✅ |
| 再試行 | 95% | 90% | ✅ |
| 境界条件 | 100% | 100% | ✅ |
| レスポンス時間 (p95) | 3.2s | <10s | ✅ |
```

## 判断ルール
- 合格基準を1つでも満たさない場合はリリースブロック
- 評価データセットは最低10件以上
- テスト環境の LLM 応答はキャッシュして安定させる（コスト節約にもなる）

## 出力形式
- 評価結果レポート（Markdown）
- 生データ（JSON）

## 注意点
- 文字一致で評価しない（LLM の出力は非決定的）
- 評価にはコスト（API 呼び出し）がかかるため、CI では頻度を制限する

## 失敗時の扱い
- 構造が不合格: プロンプトの出力形式指定を改善する
- 引用が不合格: RAG の検索精度を改善する（チャンク分割、エンベディングモデルの変更）
- レスポンス時間が不合格: MAX_TOKENS の削減、ストリーミング応答の導入を検討
