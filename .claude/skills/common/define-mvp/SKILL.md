# define-mvp

## この skill を使う場面
- 新規プロジェクトの立ち上げ時
- 機能要望が多すぎてスコープの絞り込みが必要なとき
- product-manager が MVP の境界を決定するとき

## 入力前提
- 構造化された要件ドキュメント（`clarify-product-requirements` の出力）
- ビジネスゴール（KPI、期限）

## 実行手順

### 1. 要件の棚卸し
すべての要件を一覧化し、以下の情報を付加する:
- ビジネス価値 (1-5)
- 技術リスク (1-5)
- 実装コスト (S / M / L / XL)

### 2. MoSCoW 分類
| 分類 | 基準 |
|------|------|
| **Must Have** | これがないとプロダクトが成立しない |
| **Should Have** | 重要だが、初回リリースでなくても良い |
| **Could Have** | あると嬉しいが、なくても価値は出せる |
| **Won't Have** | スコープ外として明示的に除外 |

### 3. 依存関係の整理
- Must Have 間の依存関係を整理する
- 技術的な依存（DB スキーマ → API → UI）の順序を確定する

### 4. MVP 定義書の作成
```markdown
## MVP 定義: [プロダクト名]

### ゴール
[MVP で達成する最小目標]

### Must Have
1. [機能1] — [理由]
2. [機能2] — [理由]

### Should Have (v1.1)
1. [機能A] — [理由]

### Could Have (v1.2+)
1. [機能X] — [理由]

### Won't Have
1. [機能Y] — [除外理由]

### 成功基準
- [具体的な KPI / メトリクス]

### 想定スケジュール
- MVP: [期間]
- v1.1: [期間]
```

## 判断ルール
- Must Have が 5 個以上ある場合: さらに絞り込みが必要。「これがないと1人目のユーザーが使えない」で判断
- AI 機能の扱い: AI は Must Have にするなら、フォールバック（非AI動作）も Must Have にする
- 認証: ほぼすべてのケースで Must Have。認証なしで公開する場合のみ除外可

## 出力形式
上記テンプレートに従った Markdown ドキュメント

## 注意点
- MVPは「安っぽい製品」ではなく「最小で価値を出せる製品」
- 技術基盤（認証、DB、Docker セットアップ）はユーザー向け機能ではないが、暗黙の Must Have として扱う

## 失敗時の扱い
- Must Have が多すぎて削れない: ビジネスゴールを再確認し、「1人目のユーザー」ペルソナに絞る
- 期限に間に合わない: Must Have の中で更に Phase 分けを行う
