# review-security-with-pentagi

## この skill を使う場面
- リリース前のセキュリティ検証
- security-reviewer がセキュリティスキャンを実施するとき

## ⚠️ 安全上の厳守事項
1. **本番環境に対して絶対に実行しない**
2. **ステージングまたは隔離環境に限定する**
3. 誤検知を前提とし、すべての結果を人間が再確認する
4. スキャン実行前にターゲット環境の管理者に通知する

## 入力前提
- ステージング環境の URL
- テスト用の認証情報
- 検証対象のスコープ（API エンドポイント一覧）
- `setup-pentagi-scan` skill が完了していること

## 実行手順

### 1. スキャン対象の定義
```yaml
# pentagi-scan-config.yml
target:
  base_url: https://staging.example.com
  environment: staging  # 本番は絶対禁止
scope:
  - /api/auth/*        # 認証
  - /api/llm/*         # LLM エンドポイント
  - /api/users/*       # ユーザー管理
  - /api/conversations/* # 会話管理
exclude:
  - /api/health        # ヘルスチェックは除外
```

### 2. 検証カテゴリ
| カテゴリ | 検証内容 |
|---------|---------|
| 認証 | JWT の有効性検証、トークン期限切れ、無効トークン |
| 認可 | RBAC の正確性、権限昇格の試行 |
| 入力バリデーション | SQLi、XSS、コマンドインジェクション |
| ファイルアップロード | 危険なファイル形式、サイズ制限 |
| セッション | セッション fixation、CSRF |
| LLM 固有 | プロンプトインジェクション、出力リーク、トークン枯渇 |
| CORS | 不正オリジンからのアクセス |
| 情報漏洩 | エラーメッセージのスタック内容、バージョン情報 |

### 3. PentAGI スキャン実行
```bash
# 隔離 Docker 環境で実行
docker run --rm \
  --network=isolated-network \
  -v $(pwd)/pentagi-scan-config.yml:/config/scan.yml \
  pentagi/scanner:latest \
  --config /config/scan.yml \
  --output /results/report.json
```

### 4. 結果の分析と分類
| 重要度 | 基準 | 対応 |
|--------|------|------|
| critical | データ漏洩、認証バイパス、RCE | 即修正。リリースブロック |
| high | SQLi、XSS、CSRF | リリース前に修正必須 |
| medium | 情報漏洩（バージョン露出）、セッション管理の弱点 | 次スプリントで修正 |
| low | ベストプラクティスからの逸脱 | バックログに登録 |

### 5. レポート作成
```markdown
## セキュリティスキャン結果
- 実施日: YYYY-MM-DD
- 対象環境: ステージング
- スキャンツール: PentAGI

### サマリー
| 重要度 | 件数 |
|--------|------|
| critical | 0 |
| high | 1 |
| medium | 3 |
| low | 5 |

### 発見事項
| # | 重要度 | カテゴリ | 内容 | 再現手順 | 推奨対応 |
|---|--------|---------|------|---------|---------|
| 1 | high | 入力バリデーション | ... | ... | ... |
```

### 6. 再検証
修正後に同一スキャンを再実行し、修正が有効であることを確認する。

## 判断ルール
- critical / high が 1 件でもあればリリースブロック
- 誤検知と判断した場合は理由を記録して False Positive として管理
- 検出されたが修正が難しい場合は WAF による一時緩和も選択肢

## 出力形式
- セキュリティスキャンレポート（Markdown）
- 発見事項の一覧（テーブル形式）

## 注意点
- スキャン結果は機密情報として扱い、公開リポジトリにコミットしない
- LLM 固有の攻撃面（プロンプトインジェクション）は手動テストも併用する

## 失敗時の扱い
- PentAGI が接続できない: ネットワーク設定を確認。隔離ネットワーク内でのアクセスを確認
- 誤検知が多すぎる: スキャン設定の調整、除外ルールの追加
- critical 発見: 即座に開発チームに通知し、修正を最優先とする
