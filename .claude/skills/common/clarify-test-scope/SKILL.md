# clarify-test-scope

## この skill を使う場面
- テストすべき範囲が不明確なとき
- tdd-coach がテスト計画の曖昧さを解消するとき

## 入力前提
- 機能要件と受け入れ条件

## 実行手順

### 1. テスト対象の特定
- 新規コード / 変更コードを特定する
- 影響範囲を確認する（依存しているモジュール）

### 2. テスト種別の振り分け
| 質問 | Unit | Integration | E2E |
|------|------|-------------|-----|
| 外部依存なしでテスト可能？ | ✅ | | |
| DB アクセスが必要？ | | ✅ | |
| 画面操作が必要？ | | | ✅ |
| ユーザーシナリオ全体？ | | | ✅ |

### 3. テストケースの列挙
- 正常系（Happy Path）
- 異常系（バリデーションエラー、認証エラー）
- 境界条件（空入力、最大長、特殊文字）
- AI 固有（構造検証、フォールバック動作）

## 判断ルール
- Unit テストでカバーできるなら Unit テストを優先（実行速度が速い）
- E2E テストは最小限に抑える（メンテナンスコストが高い）

## 出力形式
- テスト振り分け表（Markdown テーブル）

## 注意点
- テスト不要のケースも明示する（設定ファイルのみの変更等）

## 失敗時の扱い
- 振り分けに迷う場合: とりあえず Unit で書き、必要に応じて Integration に移動
