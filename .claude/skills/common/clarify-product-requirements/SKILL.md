# clarify-product-requirements

## この skill を使う場面
- 新機能の開発依頼を受けたとき
- 要件が曖昧で、実装に進めないとき
- product-manager が要件を整理する必要があるとき

## 入力前提
- ユーザーからの機能要望（テキスト、会話ログ、チケット）
- 既存の要件ドキュメント（あれば）

## 実行手順

### 1. 要件の収集
- ユーザーの発言から以下を抽出する:
  - 誰が（ペルソナ）
  - 何をしたいのか（機能）
  - なぜ（ビジネス価値）
  - いつまでに（期限、あれば）

### 2. 要件の構造化
以下のテンプレートに整理する:

```markdown
## 機能名: [機能名]

### 概要
[1-2文で機能の目的を説明]

### ユーザーストーリー
As a [ペルソナ],
I want [機能],
So that [ビジネス価値].

### 受け入れ条件
- Given: [前提条件]
  When: [アクション]
  Then: [期待結果]

### スコープ外
- [明示的に含めないもの]

### AI 機能の場合の追加項目
- 期待精度: [構造的に正しい / ファクトベース / 創造的]
- レスポンス時間: [ユーザーが許容できる最大秒数]
- フォールバック: [AI が応答できない場合の動作]
```

### 3. 曖昧さの特定
- 「良い」「適切な」「必要に応じて」など曖昧な表現をリストアップする
- 各曖昧表現に対して、具体的な判断基準を定義するか、ユーザーに確認する

### 4. スコープの確定
- Must Have / Should Have / Could Have / Won't Have に分類する (MoSCoW)
- MVP に含めるものを明確にする

## 判断ルール
- 要件が3文以下で表現できないほど複雑な場合: 機能を分割する
- AI 機能が含まれる場合: 必ず「期待精度」「フォールバック」を定義する
- ユーザーに確認が必要な場合: 確認事項リストを作成してから聞く（まとめて確認）

## 出力形式
上記テンプレートに従った Markdown ドキュメント

## 注意点
- 技術的な実装方針はこの skill では決めない（solution-architect が判断する）
- テスト計画はこの skill では書かない（tdd-coach が受け入れ条件からテストを導出する）

## 失敗時の扱い
- ユーザーが要件を説明できない場合: 類似サービスの例を示して比較で引き出す
- 要件が矛盾している場合: 矛盾点を明示し、トレードオフの選択肢を提示する
